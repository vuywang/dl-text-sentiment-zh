基于深度学习的中文文本情感分析系统

摘要

中文文本情感分析属于自然语言处理中的常见任务，应用场景并不抽象，商品评价、影视评论、售后反馈、社交平台留言中都能见到。文本量一大，人工逐条判断情绪倾向会变得很慢，而且主观差异明显，结果也不容易保持一致。围绕这一问题，完成了一套基于深度学习的中文文本情感分析系统。系统以浏览器访问为主要使用方式，后端采用 FastAPI，页面层使用 Jinja2 模板渲染，前端界面结合 Bootstrap、原生 JavaScript 与 ECharts 实现，数据持久化使用 SQLite 与 SQLAlchemy，模型部分采用 PyTorch、Transformers，并固定使用 google-bert/bert-base-chinese 预训练模型与 ChnSentiCorp 数据集开展二分类训练和推理。

系统不是单一的推理脚本，而是围绕完整使用流程进行了组织。页面层包含首页、单文本分析、批量分析、模型训练、模型评估和历史记录等模块。单文本分析支持中文输入校验、标签判断、置信度展示和分析历史落库；批量分析支持上传包含 text 列的 CSV 文件，完成逐行预测、统计汇总、结果预览和导出；模型训练模块允许设置 epoch、batch_size、learning_rate 与 max_length，并以后台任务方式触发训练脚本；训练完成后，系统会保存模型目录、评估指标、混淆矩阵图和损失曲线图，同时更新当前激活模型；模型评估页面可直接展示 Accuracy、Precision、Recall、F1-score 以及测试样例预测结果；历史记录页面则对分析记录、批量任务和训练任务进行集中展示。

从实现路径看，系统在应用启动时会自动完成目录创建、数据库初始化、默认模型注册与模型安全加载，接口层采用统一返回结构，异常请求会根据页面请求和 API 请求分别返回模板页面或 JSON 结果。整个系统形成了“输入文本—模型推理—结果展示—记录保存—训练更新—评估查看”的闭环流程。测试部分围绕页面访问、单文本预测、CSV 上传、训练任务创建、评估结果展示以及异常输入处理等内容展开，运行结果表明各模块能够完成既定功能，系统具备较完整的可运行性与演示性。

关键词：中文文本情感分析；深度学习；FastAPI；BERT；SQLite；情感二分类

Abstract

Chinese text sentiment analysis is a common task in natural language processing, and its practical value can be observed in product reviews, movie comments, customer feedback, and social media messages. When the amount of text keeps growing, manual judgment becomes inefficient and inconsistent. To address this problem, a browser-based Chinese text sentiment analysis system is implemented with a deep learning approach. The system uses FastAPI as the backend framework, Jinja2 for server-side page rendering, Bootstrap, native JavaScript, and ECharts for the web interface, SQLite and SQLAlchemy for persistent storage, and PyTorch with Transformers for model training and inference. The pretrained model is fixed as google-bert/bert-base-chinese, and the dataset is fixed as ChnSentiCorp for binary sentiment classification.

The system is organized as a complete application rather than an isolated inference script. It provides a home page, single-text prediction, batch CSV analysis, model training, model evaluation, and history management. The single-text module performs Chinese text validation, predicts sentiment labels, displays confidence and class probabilities, and stores records in the database. The batch module accepts CSV files containing a text column, processes them row by row, generates statistics and preview data, and exports the results as a CSV file. The training module accepts parameters such as epoch, batch size, learning rate, and max length, then starts a background training task through a subprocess. After training, the system saves the trained model, tokenizer, configuration files, evaluation metrics, confusion matrix image, and loss curve image, and updates the active model record. The evaluation page presents Accuracy, Precision, Recall, F1-score, confusion matrix, loss curve, and several sample predictions. The history page displays single prediction records, batch tasks, and training tasks in reverse chronological order.

During application startup, the system automatically creates required directories, initializes the database, registers a default model, and safely loads the active model. A unified response structure is used in API design, while page requests and API requests are handled differently in the exception layer. In this way, the full workflow of text input, model inference, result display, record persistence, model training, and evaluation visualization is completed. Functional testing and boundary-case testing show that the implemented modules can operate normally and support a relatively complete end-to-end sentiment analysis process.

Key words: Chinese text sentiment analysis; deep learning; FastAPI; BERT; SQLite; binary classification

第1章 绪论

1.1 研究背景

网络文本越来越多。短评、留言、售后反馈、商品评价这类内容更新快、数量大，人工查看的方式还能处理少量样本，一旦数据堆积起来，效率就会明显下降。尤其是中文场景，表达方式较灵活，反问、口语、省略和情绪化词语经常混在一起，单靠关键词匹配往往不够稳定。情感分析正是在这样的背景下逐步受到重视。

早期文本情感识别更多依赖人工特征设计，例如情感词典、词频统计、传统分类器组合等。这类方法有一定基础，思路也不复杂，不过一旦遇到上下文依赖、否定表达或者词义转折，效果容易波动。深度学习方法进入这一领域后，文本表示方式发生了变化。词向量、循环神经网络、卷积神经网络，再到 Transformer 结构和 BERT 预训练模型，模型对语义上下文的刻画能力逐步增强[1][2]。中文情感分析任务中，BERT 及其衍生方案已经成为较常见的实现路径。

从实际开发角度看，仅有训练代码还不够。系统是否便于运行，页面是否能展示结果，训练记录和分析记录是否能保存下来，这些问题在项目落地时都绕不开。单独的 notebook 或零散脚本可以完成实验，但不容易形成完整业务流程。于是，本课题把重点放在“可运行的系统化实现”上，将模型推理、批量分析、训练管理、评估展示和历史记录组织到同一套应用中，使文本情感分析不只停留在模型实验层面。

1.2 研究意义

研究意义主要体现在两个方面。一个是应用层面。中文情感分析能够帮助使用者快速筛选评论倾向，辅助观察某类文本的整体反馈方向，在电商评价、舆情观察、服务反馈处理等场景中都有现实价值。虽然本课题采用的是固定数据集和固定二分类任务，范围并不扩张，但这种聚焦式实现更容易形成稳定流程。系统结构完整，功能边界清楚，能够把文本分析的核心链路展示出来。

另一个是工程层面。模型训练与推理往往被看作算法问题，可一旦进入系统实现，就会涉及接口组织、页面渲染、数据表设计、文件存储、异常处理、日志记录等内容。很多时候，难点不只在模型本身，而在于如何把这些环节接起来。本课题围绕 FastAPI、Jinja2、SQLite、PyTorch、Transformers 等技术完成一体化实现，将“可训练”“可预测”“可查看记录”“可展示评估图像”结合起来，这种组织方式能够较直观地反映一个小型深度学习应用的完整开发过程。

1.3 国内外研究现状

文本情感分析研究起步较早，国外相关工作在传统机器学习阶段已经形成较成熟的分类框架，如朴素贝叶斯、支持向量机、逻辑回归等方法，常与人工构造特征、n-gram 表示和情感词典结合使用。随着深度学习的发展，基于神经网络的表示学习逐渐取代大量手工特征工程，模型开始直接从语料中学习更丰富的文本表示。Transformer 结构提出后，自注意力机制在长距离依赖建模上的优势被进一步放大，BERT 通过大规模预训练与下游微调的方式，在多项自然语言处理任务中取得了较好效果[1]。

中文情感分析的研究路径与这一趋势基本一致，只是中文分词、语义歧义、口语表达和领域差异会带来一些额外影响。国内在影评、商品评论、新闻评论等领域构建了多种中文情感数据集，也出现了大量基于 BiLSTM、TextCNN、BERT、RoBERTa 等模型的研究工作。与学术研究相比，实际系统常常更强调完整流程，也就是从数据输入到结果输出是否连贯、是否便于查看和维护。换句话说，算法精度是一部分，系统完成度也是一部分。

当前不少研究平台会进一步扩展为多模型比较、多任务支持、在线标注或可视化分析平台。这类设计覆盖范围更大，但结构也会迅速复杂化。本课题没有把范围继续铺开，而是固定为中文文本情感二分类，预训练模型固定为 google-bert/bert-base-chinese，数据集固定为 ChnSentiCorp，围绕固定任务做系统化实现。这样处理后，研究重点会更集中，项目边界更明确，也更容易贴合代码落地。

1.4 研究内容

本课题的研究内容主要包括以下几个方面。

其一，完成系统整体架构设计。系统采用单体式 Web 应用组织方式，FastAPI 既承担页面访问，也承担 API 接口处理，Jinja2 模板负责页面渲染，SQLite 负责数据保存，PyTorch 与 Transformers 负责模型训练与推理。页面与接口位于同一应用中，结构比较集中。

其二，完成核心功能实现。系统围绕首页、单文本分析、批量 CSV 分析、模型训练、模型评估和历史记录六个模块展开。单文本分析强调即时返回与历史保存，批量分析强调文件校验、逐行预测、结果导出和统计展示，模型训练强调训练参数设置和后台任务组织，评估页面强调指标与图像展示，历史记录则负责对前述操作结果进行留存与查看。

其三，完成数据库和文件存储设计。系统使用 SQLite 保存分析记录、批量任务、训练任务和模型注册信息，并使用 storage 目录统一组织数据库文件、日志文件、上传文件、导出文件、数据集缓存和模型文件。页面展示所需的混淆矩阵图、损失曲线图，也通过该目录提供静态访问。

其四，围绕项目实际完成测试与分析。测试内容不追求夸张的并发指标，而是围绕功能闭环展开，例如页面能否打开、文本预测是否返回结果、空文本是否被拦截、CSV 格式错误是否能提示、训练任务是否能创建、评估页面是否能显示指标与图像、历史记录是否能更新等。

第2章 相关技术与理论基础

2.1 FastAPI 与 Web 应用组织方式

FastAPI 是一个基于 Python 的现代 Web 框架，接口定义清晰，类型标注支持较好，开发效率也比较高[4]。本系统没有采用前后端完全分离结构，而是把页面渲染和 API 处理统一在同一应用中完成。这样一来，浏览器访问首页、分析页、训练页时，由 Jinja2 模板直接返回 HTML；页面中的按钮、表单和异步请求，再通过原生 JavaScript 调用后端 API。结构比较直接。请求路径也更容易管理。

在项目代码中，FastAPI 应用的入口位于 app/main.py。应用启动时通过 lifespan 生命周期函数完成目录创建、数据库初始化、默认模型注册和当前激活模型加载。除了普通页面路由外，系统还挂载了 /static 与 /storage 两个静态路径，前者负责 CSS、JavaScript 等资源访问，后者负责模型图片、导出文件等存储资源访问。页面请求与 API 请求的异常处理方式也被做了区分：页面异常返回 error.html，API 异常返回统一 JSON 结构。这个设计不复杂，但在实际应用里很有用，页面和接口的反馈形式被明确分开了。

2.2 Jinja2 模板、Bootstrap 与前端交互

Jinja2 是 Python 生态中常见的模板引擎，适合与服务端渲染页面结合使用。本系统的模板文件位于 app/templates 目录下，包括 base.html、index.html、predict.html、batch.html、train.html、evaluate.html、history.html 和 error.html。导航、样式引用与脚本引用集中定义在 base.html 中，其余页面通过继承模板完成内容填充。页面结构比较统一，看起来也更完整。

前端界面部分主要依赖 Bootstrap 和少量自定义 CSS。样式文件 static/css/style.css 对摘要卡片、内容面板、结果区域、表格文本溢出等进行了处理，页面整体风格较为简洁。图表展示没有单独引入重型前端框架，而是通过 ECharts 生成首页情感占比图和批量分析占比图。JavaScript 工具文件 static/js/app.js 中封装了 POST 请求、HTML 转义、百分比格式化和饼图渲染等函数。也就是说，这套前端不是单页应用，而是模板渲染页面加异步接口调用的方式，页面能打开，交互也能跑通。

2.3 SQLite 与 SQLAlchemy 持久化方案

数据库层采用 SQLite。这个选择比较明确。数据库文件直接保存在 storage/db/app.db 中，不需要额外部署数据库服务。对单机运行的文本分析系统来说，这种方式已经能够满足记录保存、任务留痕与结果查询的需要。系统没有设置复杂的事务流程，也没有多用户权限管理，SQLite 的轻量特性与项目边界是匹配的。

ORM 层使用 SQLAlchemy 2.0 版本[5]。数据库引擎在 app/core/database.py 中通过 create_engine 创建，并设置 check_same_thread=False 以适应当前应用运行方式。Base.metadata.create_all 用于应用初始化时自动建表。项目中共定义了 analysis_record、batch_task、train_task 和 model_registry 四张表，分别对应单文本分析记录、批量任务记录、训练任务记录和模型注册信息。表数量不多，不过结构已经覆盖了系统主流程。

2.4 PyTorch、Transformers 与 BERT 情感分类

深度学习部分采用 PyTorch 与 Transformers[6][7]。固定预训练模型为 google-bert/bert-base-chinese，分类任务固定为二分类。BERT 模型通过双向上下文编码获得更完整的语义表示，对中文短文本情感判断有较好的适配性[1]。项目中既包含推理逻辑，也包含训练逻辑。推理时，文本经过 tokenizer 编码后输入序列分类模型，输出 logits，再通过 softmax 转换为概率值，最终得到积极和消极两个分数。训练时则加载 ChnSentiCorp 数据集，并通过 AdamW 优化器开展多轮微调。

项目实现中还包含一个默认模型注册机制。应用初次运行时，如果数据库中不存在激活模型，系统会尝试将基础预训练模型保存到本地 current 目录并写入 model_registry 表。这样做的作用很直接：即使尚未完成微调，系统也能先具备可加载的模型来源，避免页面和接口在首次运行时处于完全不可用状态。训练完成后，新的模型目录会写入 archive 路径，并同步更新为当前激活模型。

2.5 数据集、评估指标与可视化

训练数据集固定为 ChnSentiCorp。项目代码中通过 datasets.load_dataset 加载对应数据，数据缓存目录位于 storage/datasets[8]。标签映射采用 0 表示消极，1 表示积极。模型配置中还显式设置了 id2label 与 label2id，使训练后模型在预测结果输出时能够对应中文标签。

模型评估部分采用 Accuracy、Precision、Recall 和 F1-score 四项常见指标[9]。同时，为了便于观察分类结果分布，系统还会输出混淆矩阵；为了观察训练过程中的损失变化，系统会绘制训练损失曲线与验证损失曲线。图像绘制通过 matplotlib 完成，保存后再由评估页面读取并展示。这个环节很重要。数字指标能说明分类效果，图像则更容易让使用者直观看到模型在当前训练任务下的状态。

第3章 系统需求分析

3.1 系统建设目标

系统建设目标可以概括为一句话：构建一套围绕中文情感二分类任务的可运行 Web 系统，使文本输入、批量分析、模型训练、结果评估和历史管理形成完整闭环。目标并不追求大而全，而是强调流程清楚、页面可见、结果可查、模型可切换。范围收得比较紧。

结合项目实现情况，系统需要达到以下几项要求。页面层要能在浏览器直接访问，不依赖单独前端工程；分析结果需要保存，不是一次性返回后就消失；批量分析既要支持上传，也要支持导出；训练任务不能直接阻塞页面请求，需要采用后台任务方式触发；训练完成后，新的模型信息要能注册到系统中，后续预测应自动使用当前激活模型；评估页面要能展示训练后的核心指标与图像结果。这样一来，系统才能体现出完整应用而不只是零散功能拼接。

3.2 功能需求分析

系统功能需求主要分为六部分。

首页模块负责总览展示。进入系统后，页面需要给出当前激活模型、数据库状态、分析总数、积极与消极文本数量、最近训练记录、最近分析记录以及情感占比图。这个模块本身不复杂，却承担了状态展示和入口导航的作用。

单文本分析模块负责即时推理。用户输入一段中文文本后，系统需要完成文本清洗、中文字符校验、模型推理、标签与概率计算、结果返回以及记录保存。页面上除了当前结果，还要显示最近若干条分析记录，便于观察输入与输出的对应关系。

批量分析模块负责 CSV 上传与结果导出。系统要求上传文件为 CSV 格式，并且包含 text 列。处理过程中需要逐行进行非空校验与中文文本校验，对每条文本完成推理后写出结果文件，同时统计积极与消极数量。页面还应提供任务摘要、图表展示与预览表格。

模型训练模块负责训练任务创建与查看。用户可以设置 epoch、batch_size、learning_rate 与 max_length，提交后系统创建训练任务记录，并在后台启动训练脚本。训练过程虽然不在页面中逐步直播，但任务创建和结果回写需要打通。

模型评估模块负责显示训练结果。对于最近完成的训练任务，系统需要展示 Accuracy、Precision、Recall、F1-score、混淆矩阵图、损失曲线图和测试样例预测结果。如果当前没有已完成训练任务，则页面应给出相应提示。

历史记录模块负责汇总展示。单文本分析历史、批量分析历史和训练任务历史需要按时间倒序保存并展示，使系统运行过程可追溯。这个需求在项目中并不少见，尤其是在需要演示完整流程时，历史记录会比临时输出更有价值。

3.3 非功能需求分析

非功能需求虽然不直接表现为单个按钮或接口，却影响系统整体质量。

一是可用性。页面路径应清晰，导航项固定，表单输入和按钮操作要尽量直接。系统当前采用模板渲染页面和统一样式设计，页面之间跳转简单，普通浏览器即可访问。这个层面的要求已经基本满足。

二是稳定性。应用启动时应自动创建目录、初始化数据库、准备模型加载环境；请求出错时，需要有明确反馈。项目中通过生命周期初始化和统一异常处理机制保证了这一点。页面异常时返回错误模板，接口异常时返回 JSON 结构，不会让用户只看到一串无提示的报错。

三是可维护性。项目目录按 core、models、routers、services、utils、templates 等层次组织，职责分区比较清楚。后续即便增加新的页面或新的接口，也能沿现有结构继续扩展，不必推翻已有代码。

四是可移植性。系统依赖写在 environment.yml 中，运行入口为 python run.py，数据库采用 SQLite，本地目录统一由配置类管理。只要准备好 Python 环境和模型依赖，单机条件下即可完成运行。部署过程相对简化。

3.4 可行性分析

技术上可行。系统使用的 FastAPI、Jinja2、SQLite、SQLAlchemy、PyTorch、Transformers 都属于较成熟的技术组件，资料较多，文档也比较完整。文本情感分析采用固定预训练模型与固定数据集，训练与推理链路较明确，技术路线稳定。

经济上可行。项目运行不依赖独立数据库服务器，也不需要购买额外商用平台。SQLite 本地存储、浏览器访问和单机训练降低了外部资源要求。深度学习训练在没有 GPU 的情况下会慢一些，不过系统已经允许通过较小参数进行运行和展示，基本成本可控。

操作上可行。用户打开浏览器即可访问系统，页面功能与导航都比较明确。即使不熟悉代码，也可以通过页面完成单文本分析、批量上传和训练任务发起。开发人员若需要查看数据库文件和日志文件，也能在 storage 目录中快速定位。

4.2 功能模块设计

从用户操作角度看，系统可以划分为六个功能模块。

首页模块负责总览展示。这里既展示系统状态，也承担导航入口。当前激活模型、数据库状态、统计数量、最近训练记录和最近分析记录都放在首页中，方便快速观察系统是否处于正常运行状态。

单文本分析模块负责即时预测。输入一段中文文本后，系统完成文本校验、模型加载、推理计算和结果保存，并在页面中展示标签与概率。由于页面同时显示最近十条记录，这个模块兼具“当前分析”和“短期历史查看”两层作用。

批量分析模块负责上传 CSV 文件并输出批量结果。该模块中既有文件保存逻辑，也有数据读取、字段校验、逐条推理、统计汇总和结果导出逻辑。模块链路较长。也是系统中较有代表性的功能之一。

模型训练模块负责参数输入、任务创建和训练详情查看。前端通过表单收集四项参数，接口层创建 train_task 记录，后台任务触发训练脚本，训练脚本再负责模型微调、评估计算、文件保存和数据库回写。

模型评估模块负责展示训练成果。指标数字、混淆矩阵、损失曲线和样例预测结果都集中在这一页，属于“训练之后看效果”的页面，不承担训练本身。

历史记录模块负责汇总浏览。系统通过三个页签分别展示单文本分析、批量分析与训练任务，形成对系统运行过程的集中留痕。

4.3 数据库设计

系统数据库使用 SQLite，核心数据表共四张。表数量不多，但功能边界较清楚。

analysis_record 表用于保存单文本分析结果，同时批量分析过程中也会默认把逐条预测结果写入该表。字段包括主键 id、输入文本 input_text、预测标签 predicted_label、置信度 confidence、积极概率 positive_score、消极概率 negative_score、模型名称 model_name 和创建时间 created_at。该表是情感分析结果最直接的持久化载体。

batch_task 表用于保存批量分析任务信息，包括原始文件名、保存路径、文本总数、积极数量、消极数量、结果文件路径、任务状态和创建时间。通过该表可以追踪每一次 CSV 上传任务的处理情况。

train_task 表用于保存训练任务信息。字段除了模型名称、数据集名称、epoch 数、batch 大小、学习率、max_length、状态和时间外，还包括 train_loss、val_loss、accuracy、precision_score、recall_score、f1_score、confusion_matrix_path 和 model_dir 等内容。训练结果和训练配置在这里被集中记录下来。

model_registry 表用于保存模型注册信息，包括模型名称、模型类型、模型目录、是否激活、备注和创建时间。预测服务会优先读取当前激活模型，从而决定推理时使用哪个模型目录。

4.5 接口设计说明

系统 API 接口数量不算多，但覆盖了主要业务。

POST /api/predict 用于单文本分析，请求体为 JSON，核心字段为 text，返回结果包括 predicted_label、confidence、positive_score、negative_score、model_name 和 record_id。
POST /api/batch/upload 用于上传 CSV 文件并触发批量分析，返回当前任务信息、图表数据和预览数据。
GET /api/batch/{task_id} 用于查看批量任务详情。
GET /api/batch/{task_id}/download 用于下载批量分析结果 CSV。
POST /api/train/start 用于创建训练任务，请求体包括 epoch、batch_size、learning_rate 和 max_length。
GET /api/train/{task_id} 用于查看训练任务详情。
GET /api/history/analysis、GET /api/history/batch 和 GET /api/history/train 用于查看三类历史记录。

接口层统一使用 code、message、data 三个字段组织返回结果。成功时 code 为 0，失败时 code 为非零错误码。这种设计简单，但可读性较好，前端也更容易统一处理。

4.6 关键设计说明

系统中有两个关键设计值得单独说明。

一是当前激活模型设计。预测服务并不直接写死某个模型目录，而是从 model_registry 表中读取 is_active 为 True 的记录。如果数据库中没有激活模型，则自动创建并注册默认模型。训练完成后，新模型会被注册并设为激活。这意味着预测模块与训练模块之间不是硬编码耦合，而是通过数据库状态进行衔接。

二是页面异常与接口异常分流设计。系统通过路径前缀判断请求是否为 /api/ 开头。若属于 API 请求，则返回 JSON 错误结构；若属于页面请求，则返回 error.html 模板。这样处理后，页面用户看到的是可阅读的提示页面，接口调用者拿到的是结构化错误信息，反馈形式更加合适。

第5章 系统实现

5.1 项目结构与运行环境实现

系统项目根目录下包含 app、static、scripts、storage、environment.yml、README.md 和 run.py 等内容。app 目录是主体代码区域，内部又按 core、models、routers、services、utils 和 templates 进行划分；static 存放前端资源；scripts 存放数据库初始化、训练和命令行批量分析脚本；storage 作为运行时资源目录，统一存放数据库文件、上传文件、导出文件、日志文件、模型文件和数据集缓存。

运行环境采用 conda 管理。environment.yml 中定义了环境名 zh_sentiment_fastapi，Python 版本固定为 3.14.3，依赖包括 FastAPI 0.135.3、Uvicorn 0.44.0、Jinja2 3.1.6、SQLAlchemy 2.0.49、python-multipart 0.0.26、torch 2.11.0、transformers 5.5.4、datasets 4.8.4、pandas 3.0.2、numpy 2.4.4、scikit-learn 1.8.0 和 matplotlib 3.10.8。环境文件把核心依赖集中写明，迁移和复现都会更方便一些。

项目启动入口为 run.py。该文件调用 uvicorn.run，将应用指向 app.main:app，监听地址为 127.0.0.1，端口为 8000，并开启 reload 模式。也就是说，系统默认以本地浏览器访问为主。这个实现很直接。

在 app/main.py 中，FastAPI 实例创建前定义了 lifespan 生命周期函数。应用启动时，会依次执行 ensure_directories、init_db、ensure_default_model 和 load_active_model_safely。目录和数据库会在启动时自动准备好，随后系统尽量加载当前可用模型。对于一个需要同时支持推理和训练的项目来说，这一步很关键，少了它，很多页面即便能打开，也可能在首次调用时才暴露出缺失问题。

5.2 数据库初始化与基础数据实现

数据库初始化逻辑位于 app/core/database.py 和 scripts/init_db.py。database.py 中通过 create_engine 构建 SQLite 连接，SessionLocal 用于生成数据库会话，init_db 函数通过导入 app.models.db 并执行 Base.metadata.create_all 完成建表。这意味着数据库表结构并不依赖手工执行 SQL 文件，而是在应用启动和初始化脚本运行时自动生成。

四张核心数据表的 ORM 模型分别位于 app/models/db/analysis_record.py、batch_task.py、train_task.py 和 model_registry.py 中。字段设计与系统主流程对应得比较紧。analysis_record 保存预测结果明细，batch_task 保存批量任务摘要，train_task 保存训练任务与评估指标，model_registry 保存模型文件来源和激活状态。表之间没有强制外键绑定，更多体现为逻辑关联。这样设计虽然简单一些，但对当前项目已经足够。

默认模型注册逻辑位于 app/services/model_service.py。系统首先查询 model_registry 表中的激活模型。如果已经存在，就直接返回；如果没有，则进一步检查历史记录中是否已经有其他模型存在；若还是没有，系统会将基础预训练模型保存到 storage/models/current/bert-base-chinese-base，并创建一条模型注册记录，模型名为 bert-base-chinese-base-unfinetuned，模型类型为 base，激活状态为 True。这个过程避免了系统在第一次运行时处于“没有任何模型来源”的状态。

从开发经验看，这种“默认模型先兜底，微调模型再替换”的思路比较实用。系统一开始就能预测，哪怕效果未必最佳，至少流程能跑起来；完成训练后，激活模型再切换到微调版本，逻辑上也比较顺。

5.3 首页与导航页面实现

页面公共骨架定义在 app/templates/base.html。该模板引入 Bootstrap、ECharts、自定义样式和通用 JavaScript 文件，并定义顶部导航栏。导航项包括首页、单文本分析、批量分析、模型训练、模型评估和历史记录六项。各页面通过继承 base.html 保持统一结构，用户在浏览时不会出现明显的风格割裂。

首页实现位于 page_router.py 中的 index_page 函数和 index.html 模板。路由函数会读取当前激活模型、数据库状态、最近训练任务、最近分析记录和情感统计信息，再统一传入模板。index.html 页面上方显示系统简介，紧接着通过五个摘要卡片展示当前激活模型、数据库状态、分析总数、积极数量和消极数量。下半部分左侧为情感占比饼图，右侧为最近训练记录，下方为最近分析记录表格。

5.4 单文本分析功能实现

单文本分析页面由 page_router.py 中的 predict_page 函数返回 predict.html 模板。页面左侧是文本输入区，右侧是结果展示区，下方为最近十条分析记录表格。用户在 textarea 中输入中文文本后，点击“开始分析”，前端脚本会将文本内容以 JSON 形式发送到 POST /api/predict 接口。

接口层位于 app/routers/predict_router.py。路由函数接收 PredictRequest 请求模型，其中 text 字段通过 Pydantic 设定了最小长度约束。进入业务层后，predict_service.py 中的 predict_and_save 函数先调用 require_text 对文本进行清洗和校验。该函数位于 text_utils.py 中，主要完成两件事：一是通过正则将多余空白折叠为单个空格并去首尾空白，二是要求文本必须包含中文字符，同时排除乱码字符“�”。若文本为空或不含中文，则直接抛出异常。

校验通过后，服务层调用 model_service.py 中的 predict_text 完成真正的推理逻辑。该函数会先确保当前激活模型已经加载，然后使用 AutoTokenizer 进行编码，设置 truncation、padding 和 max_length，接着调用 AutoModelForSequenceClassification 得到 logits，并经过 softmax 计算负向和正向两类概率。项目中约定 positive_score 对应索引 1，negative_score 对应索引 0，若 positive_score 大于等于 negative_score，则预测标签为“积极”，否则为“消极”。最后将置信度和两个概率保留六位小数后返回。

预测完成后，系统并不会只把结果回传给页面，而是会创建 AnalysisRecord 记录写入数据库。记录内容包括原始输入文本、预测标签、置信度、概率值和模型名。这样一来，页面刷新之后，分析历史依然可以查看。对系统化项目来说，这一步很重要，不然后续首页统计和历史记录模块都无法成立。

predict.html 的前端脚本会根据接口返回值动态渲染结果区域，标签以大字体展示，概率与置信度以下拉形式排列。若接口返回失败，则结果区域显示警告信息，状态栏同步提示错误原因。

5.5 批量分析功能实现

批量分析页面位于 batch.html。页面顶部说明 CSV 文件必须包含 text 列，中部提供文件上传表单，处理完成后显示任务统计、积极/消极占比图和结果预览表格，页面下方还会展示最近批量任务列表。交互上，用户选择文件并提交后，前端使用 FormData 将文件发送到 POST /api/batch/upload 接口。

文件处理逻辑集中在 app/services/batch_service.py。接口拿到 UploadFile 后，先调用 save_upload_file 将文件保存到 storage/uploads 目录。文件名不是原样照搬，而是通过 safe_filename 进行清洗和时间戳拼接，这样可以减少文件名冲突和特殊字符带来的路径问题。保存完毕后，系统立即创建一条 BatchTask 记录，状态先设为 running。

进入数据读取阶段后，系统调用 read_csv_compatible 读取 CSV 文件。该函数会优先尝试 utf-8-sig，再尝试 utf-8，若两种编码均失败，则抛出“CSV 编码需为 utf-8 或 utf-8-sig”的异常。随后，ensure_text_column 检查数据表中是否存在 text 列，如果缺失则直接报错。逐行处理时，代码以 start=2 的方式记录行号，也就是默认第一行为表头，因此实际报错行号会更符合 CSV 阅读习惯。若某行 text 为空，则给出“第 n 行 text 不能为空”的提示。

每一条文本通过 require_text 清洗后，系统调用 predict_text 完成推理，并将结果一并写入 analysis_record。此处有个细节。批量分析不仅会生成汇总任务，也会默认写入逐条分析历史，这使得首页统计和历史记录页面能够同步看到批量处理带来的结果变化。所有记录处理完成后，系统把 text、predicted_label、confidence、positive_score、negative_score 五列写入新的结果文件，文件保存到 storage/exports，命名格式为 batch_result_任务ID_时间戳.csv。

任务完成后，BatchTask 的 total_count、positive_count、negative_count、result_file_path 和 status 会被更新为最终值。前端拿到返回结果后，一方面用 ECharts 绘制本次批量分析饼图，另一方面生成预览表格，并将下载按钮链接设置为 /api/batch/{task_id}/download。整个流程是闭合的。

这个模块的工程意味比较浓。上传、保存、读取、校验、推理、落库、导出、回显，环节一个都不少。也正因为如此，批量分析模块在整套系统里占据了比较重要的位置。

5.6 模型训练功能实现

模型训练页面由 train.html 提供。页面上方写明训练任务固定为 ChnSentiCorp 数据集和 google-bert/bert-base-chinese 预训练模型，表单中可设置 epoch、batch_size、learning_rate 和 max_length 四项参数。各参数不仅在前端表单中给出范围，后端 Pydantic 请求模型 TrainStartRequest 也作了约束：epoch 在 1 到 20 之间，batch_size 在 1 到 128 之间，learning_rate 大于 0 且不超过 1，max_length 在 16 到 512 之间。前后两层都有限制，非法参数更难直接穿透到训练脚本中。

训练任务创建接口位于 app/routers/train_router.py。前端提交表单后，接口调用 start_training_task 创建 TrainTask 记录，并为模型命名为 bert-base-chinese-finetuned-时间戳，状态置为 running。创建完数据库记录后，不是直接在当前请求里训练，而是通过 BackgroundTasks 调用 run_training_subprocess。这样处理的目的很明确：避免页面请求长时间阻塞。

run_training_subprocess 会构造训练命令，调用 scripts/train_model.py，并把标准输出和标准错误写入 storage/logs/train_task_{id}.log。训练脚本内部首先通过 argparse 接收任务 ID、epoch、batch-size、learning-rate、max-length 和 model-name 等参数，然后初始化数据库连接，找到对应训练任务记录。数据集加载使用的是 settings.DATASET_HF_ID，即 lansinuote/ChnSentiCorp，缓存目录为 storage/datasets。代码默认读取 train 划分作为训练集，若数据集中包含 validation 划分则优先使用，否则使用 test 作为评估集。

模型部分采用 AutoTokenizer 和 AutoModelForSequenceClassification，分类标签数固定为 2，并显式设置 id2label 与 label2id 为“消极”和“积极”。训练采用 DataLoader 组织批次，优化器为 torch.optim.AdamW。每一轮训练完成后，系统都会调用 evaluate 函数计算验证损失、准确率、精确率、召回率、F1 值和混淆矩阵，并把 train_loss 与 val_loss 记录到列表中，以便后续绘制损失曲线。

训练结束后，脚本会把 tokenizer 和模型权重保存到 storage/models/archive/task_任务ID_时间戳 目录中，同时生成 confusion_matrix.png、loss_curve.png、metrics.json 和 train_config.json。随后，训练任务表中的 train_loss、val_loss、accuracy、precision_score、recall_score、f1_score、confusion_matrix_path、model_dir、status 和 finished_at 等字段被更新。接着，系统将 model_registry 表中旧模型的 is_active 全部置为 False，再插入新模型注册记录，并将其设为当前激活模型。也就是说，训练完成后的模型不仅被保存下来，也会立即接管后续预测。

训练模块中还有一个容易被忽略的点。matplotlib 在脚本头部被设置为 Agg 后端，这意味着图像绘制不依赖图形界面环境，适合直接在脚本和服务环境中运行。同时，代码设置了中文字体候选列表，用于保证混淆矩阵和损失曲线中的中文标签显示正常。细节不算多，却很有针对性。

5.7 模型评估页面实现

模型评估页面由 evaluate_page 函数返回 evaluate.html 模板。页面进入时，系统会查询最近一条状态为 completed 的训练任务，并把该任务的评估指标传递给模板。如果查询不到已完成训练任务，则页面显示“暂无已完成训练任务，请先在模型训练页面启动并完成一次训练”的提示。这一逻辑与系统实际状态一致，不会空展示不存在的结果。

5.9 关键代码逻辑与异常处理实现

除了页面和业务流程外，系统还有几处关键代码逻辑值得说明。

一是统一返回结构。utils/response.py 中定义了 success 和 fail 两个函数，成功时返回 code、message 和 data，失败时返回 code、message 和 data=null。路由层基本都围绕这一结构返回结果，前端脚本也据此判断请求成功与否。结构简单，但接口风格保持得比较一致。

二是日志处理。core/logger.py 中通过 RotatingFileHandler 把日志写入 storage/logs/app.log，单个文件最大 5MB，保留 5 个备份。同时日志也输出到控制台，方便开发时直接查看。训练模块则单独把每个任务的输出写入 train_task_{id}.log，定位训练错误会更方便。

三是异常分流。main.py 中定义了 HTTPException、RequestValidationError 和普通 Exception 三类异常处理器，并通过 _is_api_request 判断当前请求是否属于 /api/ 前缀。API 请求失败时返回 JSON 错误信息；页面请求失败时返回 error.html 模板，并在页面中提示查看日志文件。这种处理很符合当前系统形态。

四是模型缓存加载。model_service.py 中通过全局变量和线程锁缓存 tokenizer、模型实例、设备信息和激活标记，避免每次请求都重复加载模型。加载时会根据 model_registry 当前激活记录决定模型来源，若本地模型目录中存在 config.json，则从本地加载，否则直接回退到预训练模型名称。这个设计兼顾了运行效率与容错性。

从系统运行情况来看，这些基础实现虽然不显眼，却在很大程度上保证了系统的完整性。页面能打开、接口能返回、日志能定位、模型能切换，很多时候正是这些环节在托住整个应用。

6.3 异常场景与边界场景测试

异常测试部分更能反映系统是否具备基本防护能力。

在单文本分析页输入空文本后，前端会直接提示“请输入文本内容”，请求不会继续发送。若通过接口提交空字符串或非法字符串，后端 require_text 也会再次校验，并返回“文本内容不能为空”或“请输入包含中文字符的文本”等错误信息。也就是说，系统对空输入和非中文输入都设置了防线。

【插图建议：图6-5 单文本分析输入空文本时的提示截图】

在批量分析模块中，若上传非 CSV 文件，save_upload_file 会直接拒绝保存，并返回“只支持上传 CSV 文件”；若 CSV 编码不是 utf-8-sig 或 utf-8，则 read_csv_compatible 会提示编码错误；若 CSV 不包含 text 列，ensure_text_column 会返回“CSV 文件必须包含 text 列”；若 text 列中某一行为空，则会返回具体的行号提示。这样的错误反馈比较具体，不是笼统一句“上传失败”。

【插图建议：图6-6 上传缺少 text 列的 CSV 后的错误提示截图】

训练模块的边界场景主要体现在参数约束上。若 epoch、batch_size、learning_rate 或 max_length 超出允许范围，Pydantic 校验不会通过，接口返回请求参数校验失败。若训练脚本执行过程中发生异常，run_training_subprocess 会将对应 train_task 状态更新为 failed，并写入 finished_at，日志文件中保留训练输出，用于后续定位问题。这种失败回写机制能够防止任务长期停留在 running 状态。

评估页面的边界场景则更直观。如果不存在已完成训练任务，页面会显示提示信息而不是尝试加载不存在的图像文件。若图片路径异常，模板层也会通过条件判断避免空链接直接渲染。页面反馈相对温和。

6.4 测试结果分析

综合测试结果可以看出，系统已经完成了从模型加载、单文本推理、批量分析、训练任务创建、评估结果展示到历史记录查看的主要流程。各模块之间不是孤立存在的。单文本和批量分析会写入 analysis_record，训练任务结束后会更新 train_task 和 model_registry，评估页面又会读取最新训练结果并进行展示，首页统计与历史记录也会同步变化。数据流动是连续的。

系统也表现出一些客观特点。优点在于结构清晰、页面完整、运行链路闭合，用户不需要切换多套工具即可完成主要操作。训练和推理都被纳入统一系统中，这一点比较实用。另一方面，当前实现仍有边界：训练过程没有实时进度条，评估页面默认只展示最近一条已完成训练任务，任务量继续增加后历史记录页面也缺少搜索和分页；模型训练在没有 GPU 的情况下耗时会较明显，体验上会受到一定影响。这些问题并不影响系统主流程成立，但在后续扩展中值得继续处理。

6.5 本章小结

本章围绕系统真实功能完成了测试与分析。测试结果表明，系统页面能够正常访问，单文本分析、批量分析、模型训练、模型评估和历史记录等模块均可按预期运行；空文本、错误编码、缺失字段、非法参数等异常场景也能够得到明确反馈。整体上看，系统已经具备较完整的运行能力。

第7章 结论与展望

本课题围绕中文文本情感二分类任务，完成了一套基于深度学习的 Web 系统实现。系统以后端 FastAPI 为组织中心，结合 Jinja2 模板、Bootstrap、原生 JavaScript、ECharts、SQLite、SQLAlchemy、PyTorch 和 Transformers，构建了首页、单文本分析、批量分析、模型训练、模型评估和历史记录等模块。项目实现不是停留在单个脚本层面，而是形成了较完整的使用流程。文本输入后可以得到预测结果，批量文件上传后可以导出 CSV，训练任务完成后能够生成指标与图像，系统还会把结果保存在数据库中并集中展示。整个过程是连通的。

从实现效果看，系统较好地完成了预定目标。应用启动时自动创建目录和数据库，默认模型可自动注册，异常请求能够给出对应反馈；预测服务和训练服务通过 model_registry 建立起联系，模型切换逻辑也较清晰。页面结构统一，操作路径简短。对于中文情感分析这一固定任务而言，系统已经能够承担基本的训练、推理和展示工作。

当然，当前系统还存在可以继续完善的地方。其一，任务类型仍然固定为二分类，数据集也固定为 ChnSentiCorp，后续可以考虑扩展为多数据集切换或更细粒度情感类别，但这需要同步调整训练配置、标签体系和页面展示逻辑。其二，训练过程当前采用后台任务加子进程方式，能够避免阻塞请求，不过实时反馈仍然有限，后续可加入训练进度轮询或更细的日志可视化。其三，历史记录页面还没有搜索、筛选和分页功能，当记录继续累积后，页面可读性会受到影响。其四，模型推理目前依赖完整 BERT 序列分类模型，在资源受限环境下仍然会有一定开销，后续可以考虑蒸馏模型或轻量化部署方案。

整体来看，这套系统已经较完整地把中文文本情感分析任务从“模型代码”推进到了“系统应用”层面。结构不夸张，流程是完整的。对于后续继续扩展功能、优化体验或替换模型，也保留了较清晰的改进空间。
